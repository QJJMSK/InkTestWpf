1..Net の既知の問題

1.1 NetFx のインストール

1.1.1 .NET Framework 4.5 Language Pack は Windows 8 では使用できない

.NET Framework 4.5 はオペレーティング システムのコンポーネントであるため、Windows 8 オペレーティング システムには .NET Framework 4.5 Language Pack をインストールできません。.NET Framework 4.5 Language Pack は、サポートされている以前のバージョンの Windows にはインストールできます。

この問題を解決するには:

Windows 8 オペレーティング システムの Language Pack をダウンロードします。または、ローカライズ版の Windows 8 をインストールし、.NET Framework 4.5 のローカライズされたリソースを取得します。

1.1.2 .NET Framework 4.5 のインストールの失敗またはキャンセルにより、アプリケーション プールが 2.0 に戻る

.NET Framework 4 から .NET Framework 4.5 RTM リリースにアップグレードしようとしていて、インストールが取り消されるか失敗すると、.NET Framework が Version 4 に戻ります。ただし、ASP.NET 4 も登録されず、IIS のすべてのアプリケーション プールが .NET Framework 2.0 を対象とするように設定されます。

この場合、ASP.NET 4 に基づいている Web アプリケーションを参照すると、"compilation" 要素の "targetFramework" 属性が認識されないことを報告する構成エラーが発生します。

この問題を解決するには:

この問題を回避するには:

  1. 次のオプション A またはオプション B を完了して、間違った登録を修正します (両方を行う必要はありません)。
    • 次のコマンドを使用して、ASP.NET IIS 登録ツール (aspnet_regiis) を実行して ASP.NET 4 を登録します。
      • %windir%\Microsoft.Net\Framework(64)\v4.0.30319\aspnet_regiis.exe -iru -enable

      - または -

    • インストール プログラムを再実行し、[修復] オプションを選択して、.NET Framework 4 Extended Profile を修復します。

  2. また、アップグレードが失敗する前に ASP.NET 4 を対象とするカスタム アプリケーション プールがあった場合は、IIS マネージャー (inetmgr.exe) を使用して ASP.NET 4 を対象とするようにプールを手動で再設定します。

1.2 NetFx のアンインストール

1.2.1 Windows 8 または Windows Server 2012 から .NET Framework 4.5 を削除すると、ASP.NET 2.0 および 3.5 が動作しない

Windows 8 および Windows Server 2012 では、ASP.NET 2.0 および 3.5 が動作するには ASP.NET 4.5 の機能が有効になっている必要があります。.NET Framework 4.5 を削除または無効にした場合、ASP.NET 2.0 および 3.5 のアプリケーションは動作しなくなります。

この問題を解決するには:

Windows 8 の場合

コントロール パネルでの ASP.NET 4.5 の機能の有効化:

  1. コントロール パネルを開きます。
  2. [プログラム] を選択します。
  3. [プログラムと機能] で、[Windows の機能を有効化または無効化] を選択します。
  4. .NET Framework 4.5 Advanced Services ノードを展開します。
  5. [ASP.NET 4.5] チェック ボックスをオンにします。
  6. [OK] をクリックします。

Windows Server 2012 の場合

IIS ラーニング センターの「IIS 8.0 Using ASP.NET 3.5 and ASP.NET 4.5 (ASP.NET 3.5 および ASP.NET 4.5 を使用する IIS 8.0)」を参照してください。

1.3.8 Windows Communication Foundation (WCF)

1.3.8.1 WCF 4.5 での既存の XML シリアル化コード実行時の問題

Windows Communication Foundation (WCF) 4.5 では、XmlSerializer クラスは C# コンパイラへの依存関係を取り除くよう最適化されていました。この変更は、コールド スタート シナリオでパフォーマンスの向上を大幅に向上させます。ただし、XML シリアル化コードを WCF 4 でコンパイルし、WCF 4.5 で実行する場合に問題が発生する可能性があります。

この問題を解決するには:

WCF 4.5 で既存の XML シリアル化コードを実行する際に問題が発生した場合は、次の構成要素を使用して WCF 4 の XmlSerializer の動作に戻してください。

<configuration>
   <system.xml.serialization>
      <xmlSerializer useLegacySerializerGeneration="true"/>
   </system.xml.serialization>
</configuration>

1.3.10 Windows Workflow Foundation (WF)

1.3.10.1 Workflow Identity パラメーターを指定してワークフローを使用するとホストの起動に失敗する

.NET Framework 4.5 には、SQL ワークフロー インスタンス ストアに永続化された新しいワークフロー パラメーター Workflow Identity が含まれています。このパラメーターを含むワークフローを使用し、その値が null ではない場合は、値を格納できるようにインスタンス ストアを更新する必要があります。更新しないと、ホストは起動できません。

この問題を解決するには:

以下の場所にあるスクリプトを実行して、SQL ワークフロー インスタンス ストアを更新します。

%windir%\Microsoft.NET\Framework\<バージョン>\SQL\<言語>\SqlWorkflowInstanceStoreSchemaUpgrade.sql

1.3.10.2 ワークフロー デザイナーの一部の新機能により、既存のソリューションで問題が発生する可能性がある

.NET Framework 4.5 では、ワークフロー デザイナーに以下の変更が加えられています。

  • 複数のアクティビティを選択した状態で、ショートカット メニューを使用できるようになりました。
  • アクティビティ デザイナーでは、ドラッグ可能な領域がデザイナー全体ではなく、デザイナーのヘッダーになりました。FlowDecision および FlowSwitch デザイナーでは、ドラッグ可能な領域はこれまでと変わらずデザイナー全体です。
  • Ctrl キーを押した状態での複数選択の動作変更: .NET Framework 4 では、選択はマウス ダウンでキャンセルされます。.NET Framework 4.5 RC では、選択はマウス アップでキャンセルされます。選択は、今までどおりマウス ダウンで行われます。
  • アクティビティ デザイナーがアクティビティのデリゲートをサポートするようになりました。詳細については、MSDN ライブラリの「How To: Define and consume activity delegates in the Workflow Designer (方法: ワークフロー デザイナーでアクティビティのデリゲートを定義および使用する)」を参照してください。
  • 変数および引数デザイナーが、ショートカット メニューをサポートするようになりました。ショートカット メニューには、変数や引数の削除、注釈の処理などの項目があります。詳細については、MSDN ライブラリの「How To: Add comments to a workflow in the Workflow Designer (方法: ワークフロー デザイナーでワークフローにコメントを追加する)」を参照してください。
  • 以前の MetadataStore クラスでは、ジェネリック型定義のジェネリック プロパティでの属性の追加はサポートされていませんでした。これがサポートされるようになりました。
  • ModelItemDictionary クラスが null キーをサポートするようになりました。
  • 以前の ModelItem.Parents プロパティには、モデル ツリー内にない項目が含まれていました。現在は、ModelItem.Parents のすべてのオブジェクトが、モデル ツリー内にある有効なオブジェクトです。
  • 以前は、Switch アクティビティのデザイナーで Case を選択すると、キーボード フォーカスが選択されている Case のアクティビティに移動し、アクティビティが強調表示されました。現在は、アクティビティは強調表示されますが、キーボード フォーカスは変更されません。
  • 以前は、式テキスト ボックスに 1 文字だけを入力すると、デザイナーは値を Char 型の定数として扱いました。現在は、このような式は変数名または引数名として扱われます。
  • 以前は、型ブラウザーでは、参照されていないアセンブリに対して依存関係のある参照されたアセンブリ内の型が表示されていました。現在は、参照されていないアセンブリに対して依存関係のある型は、依存アセンブリが参照されるまで表示されません。
  • 次の型とメンバーは、使用されなくなりました。
    • System.Activities.Presentation.Services.ModelChangedEventArgs クラスのほとんどのプロパティは使用されなくなりました。使用されなくなったプロパティの代わりに ModelChangeInfo プロパティを使用します。
    • System.Activities.Presentation.DragDropHelper クラスでは、新しい複数選択機能によって以下のメソッドは使用されなくなっています。
      • DoDragMove(WorkflowViewElement, Point) - 代わりに DoDragMove(IEnumerable<WorkflowViewElement>, Point) を使用します。
      • GetDroppedObject(DependencyObject, DragEventArgs, EditingContext) - 代わりに GetDroppedObjects(DependencyObject, DragEventArgs, EditingContext) を使用します。
      • GetDraggedModelItem(DragEventArgs) - 代わりに GetDraggedModelItems(DragEventArgs) を使用します。
      • GetCompositeView(DragEventArgs) - 代わりに GetCompositeView(WorkflowViewElement) を使用します。
      • SetDragDropCompletedEffects(DragEventArgs, DragDropEffects) - 代わりに SetDragDropMovedViewElements(DragEventArgs, IEnumerable<WorkflowViewElement>) を使用します。
      • GetDragDropCompletedEffects(DataObject) - 代わりに GetDragDropMovedViewElements(DataObject) を使用します。

この問題を解決するには:

上で説明した各問題の回避策を使用してください。

2..Net の保守

2.1 Windows 8 にアップグレードしても .NET Framework 4 Language Pack が更新されない

Windows 7 オペレーティング システムから Windows 8 にアップグレードした場合、以前コンピューターにインストールした .NET Framework 4 Language Pack は削除も更新もされません。この問題は、アップグレードされた Windows 8 の言語と一致していない Language Pack に影響します (英語は含まれません)。たとえば、Windows 7 Ultimate の英語エディションと .NET Framework 4 のドイツ語の Language Pack があり、システムを Windows 8 の英語エディションにアップグレードした場合、ドイツ語の Language Pack はシステム上に残りますが、.NET Framework 4.5 には更新されません。

この問題を解決するには:

Windows 8 にアップグレードする前に .NET Framework 4 Language Pack をアンインストールするか、アップグレード後に適切な Windows 8 Language Pack をインストールします。
